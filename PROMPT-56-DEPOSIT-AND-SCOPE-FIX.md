# PROMPT-56: Deposit Settings + Correct Deposit Terms + AI Scope Fix

## Problem Statement

Three issues, all active in production:

**1. Deposit terms are hardcoded and wrong**
The client-facing proposal PDF says "50% deposit due at signing. Remaining 50% due upon completion." This is wrong for CreateStage — the correct policy is 50% of labor + 100% of materials up front. Two quotes went out with wrong deposit language. Money is on the line.

Worse: the deposit policy is hardcoded in the PDF generator. Different shops have different policies. A shop that takes 50% flat shouldn't have to live with CreateStage's terms baked in.

**2. Consumables not in deposit calculation**
When computing the deposit amount, consumables must be included in the materials portion. Right now the deposit language is just a text string — there's no dollar calculation at all. When we add deposit settings, consumables (alongside materials, hardware, shop stock) must feed the materials deposit total.

**3. AI scope generation silently failing**
`generate_client_scope()` in `backend/pdf_generator.py` does `from .claude_client import call_fast` — a relative import that fails because `pdf_generator.py` is a top-level module, not a sub-package. The `ImportError` is silently caught. Fallback returns the raw job description. Client proposals show the user's unformatted description instead of professional AI-written prose.

## Acceptance Criteria

1. The `User` model has two new fields: `deposit_labor_pct` (default 50, integer) and `deposit_materials_pct` (default 100, integer)
2. Both fields are readable and writable via the existing `/profile` GET and PUT endpoints — no new endpoints needed
3. The shop PDF (generated by `generate_quote_pdf`) shows payment terms that use the shop's actual deposit settings, with dollar amounts calculated from the quote
4. The client proposal PDF (generated by `generate_client_pdf`) shows payment terms that use the shop's actual deposit settings, with dollar amounts calculated from the quote
5. The deposit materials dollar amount includes: material_subtotal + hardware_subtotal + consumable_subtotal + shop_stock_subtotal (all × markup multiplier)
6. The deposit labor dollar amount is: labor_subtotal × markup multiplier × deposit_labor_pct / 100
7. At CreateStage's defaults (50% labor, 100% materials), the terms read something like: "Deposit due at signing: $X,XXX (50% of labor) + $X,XXX (100% of materials) = $X,XXX total deposit. Remaining labor balance of $X,XXX due upon completion."
8. A shop with 50% labor / 50% materials would see their own numbers calculated correctly
9. `generate_client_scope()` uses `from backend.claude_client import call_fast` — the AI scope generates professional prose instead of copying the raw job description
10. All existing tests pass

## Decomposition

### Chunk 1: Database — add deposit fields to User model

In `backend/models.py`, add two columns to the `User` class:
- `deposit_labor_pct` — Integer, default 50 (percentage of labor charged at signing)
- `deposit_materials_pct` — Integer, default 100 (percentage of materials charged at signing)

Then generate a new Alembic migration that adds both columns with their defaults. The migration must be non-destructive — existing rows get the default values.

### Chunk 2: API — expose deposit fields in profile endpoints

In `backend/routers/auth.py`:
- Add `deposit_labor_pct: Optional[int] = None` and `deposit_materials_pct: Optional[int] = None` to `ProfileUpdate`
- Add `deposit_labor_pct: int` and `deposit_materials_pct: int` to `UserResponse`
- Make sure `_user_to_response()` (or however the user dict is built) includes both fields

No new endpoints. The existing `GET /profile` and `PUT /profile` handle this automatically via the existing update loop.

### Chunk 3: Pass deposit settings through to PDF generator

In `backend/routers/pdf.py`, the `user_profile` dict is built from the current user's data before calling any PDF generator. Add `deposit_labor_pct` and `deposit_materials_pct` to that dict, reading from `current_user` with fallback defaults (50 and 100).

### Chunk 4: Fix the AI scope import

In `backend/pdf_generator.py`, inside `generate_client_scope()`:

Change:
```
from .claude_client import call_fast
```
To:
```
from backend.claude_client import call_fast
```

This single change makes AI scope generation work. No other logic changes needed in this function.

### Chunk 5: Fix deposit terms in shop PDF (`generate_quote_pdf`)

`generate_quote_pdf()` receives `user_profile` as a parameter. Read `deposit_labor_pct` and `deposit_materials_pct` from it (default to 50 and 100 if missing).

Calculate the deposit amounts from the quote data:
- Materials deposit = (material_subtotal + hardware_subtotal + consumable_subtotal + shop_stock_subtotal) × markup_multiplier × (deposit_materials_pct / 100)
- Labor deposit = labor_subtotal × markup_multiplier × (deposit_labor_pct / 100)
- Total deposit = materials deposit + labor deposit
- Remaining = total − total deposit

Replace the hardcoded payment terms text with a dynamically built string that shows:
- The deposit_labor_pct and deposit_materials_pct percentages
- The calculated dollar amounts for each
- The total deposit
- The remaining balance

Example for 50% labor / 100% materials:
```
Deposit due at signing: $2,450 (50% of labor) + $3,200 (100% of materials) = $5,650
Remaining balance due upon completion: $2,450
```

The exact format is up to the implementer — it just needs to be clear, show the numbers, and fit the existing PDF layout.

### Chunk 6: Fix deposit terms in client proposal PDF (`generate_client_pdf`)

Same logic as Chunk 5, but for `generate_client_pdf()`. The client PDF has a Terms & Conditions section with a terms list — replace the deposit term entry with the same dynamically calculated text.

Note: the client PDF already has `user_profile` passed in. Read `deposit_labor_pct` and `deposit_materials_pct` from it the same way.

### Chunk 7: Update any tests asserting old deposit text

Search `tests/` for `"50% deposit"` and `"Remaining 50%"`. If any test asserts the old hardcoded text, update the assertion to check for the presence of the deposit percentages or calculated amounts rather than the exact old string. The test should verify the concept (deposit terms present and correct) not the exact legacy phrasing.
